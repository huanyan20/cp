## 1. 專案概覽

本專案是一套**強化學習驅動的台灣科技股自動投資組合管理系統**，完整涵蓋：

```
研究階段         → 生產階段
────────────       ─────────────
資料抓取          每日自動執行
  ↓               evaluate_portfolio.py（模型預測信號）
特徵工程             ↓
  ↓              signal.json（目標持倉）
RL 訓練              ↓
  ↓              trade_guard.py（風控驗證）
Walk-Forward         ↓
驗證              cmoney_rpa.py（大富翁虛擬下單）
  ↓
Promotion Gate
（7 項品質關卡）
```

**股票宇宙（Universe）**：台灣科技 + OTC 電子股，約 45 支（`stock_universe.py`）。

**強化學習演算法**：PPO（Proximal Policy Optimization）及 SAC（Soft Actor-Critic），來自 `stable-baselines3`。

## 3. 核心概念：強化學習投資組合

### 3.1 問題定義

將投資組合管理建模為 **MDP（Markov Decision Process）**：

| 元素 | 定義 |
|---|---|
| **State（狀態）** | 每支股票過去 20 天的市場特徵 + 帳戶特徵（現金比、DD、持倉等） |
| **Action（動作）** | 各股票的目標持倉權重（logits → softmax → top-k） |
| **Reward（獎勵）** | 混合型：報酬率 + Sortino + 超越基準 - 交易成本 - 換手率 - Drawdown penalty |
| **Terminal** | 一個 Walk-Forward OOS 期間走完 |

### 3.2 動作空間（Action Space）

```python
# Long-only + Cash mode（預設）
action_space = Box(low=-5.0, high=5.0, shape=(num_stocks + 1,))
```

動作轉換流程：
```
raw logits
  → softmax (temperature=0.5)
  → top-k 篩選（預設 k=5）
  → 股票權重 + 現金權重（softmax 最後一個 logit）
  → 執行交易
```

### 3.3 Reward 設計

```python
reward = (
    0.4 × softsign(log_return × 100)            # 日報酬
  + 0.3 × softsign(sortino) × capital_util      # Sortino（調整槓桿）
  + 0.3 × softsign(超越基準日報酬 × 100)         # 跑贏前3強股（20天動量）
  - 5.0 × trade_cost                            # 交易成本懲罰
  - 1.0 × turnover / 2.0                        # 換手率懲罰
  - 0.8 × max(0, drawdown - 3%)                # R4: Drawdown 懲罰（0.3→0.8，緩衝 5%→3%）
  - 1.0 × stock_exposure × (dd - 8%)           # R4: 大跌曝險懲罰（門檻 10%→8%，係數 0.5→1.0）
  + 0.2 × cash_ratio × (dd - 8%)               # R4: 新增防禦現金獎勵（dd>8% 且 cash_action 啟用）
).clip(-1, 1)
```

> **R4 調整說明（2026-06-08）**：實驗數據顯示 MDD 幾乎全累積在 2025H1 熊市期間，
> SAC/enabled 在該期平均現金僅 7.38%，原因是舊版 `LAMBDA_DRAWDOWN=0.3` 讓單日小漲
> 就能蓋過回撤懲罰，模型沒有學到主動降曝險的強信號。R4 將 DD 懲罰放大 2.7×，
> 並新增防禦現金獎勵，使回檔期間降低曝險、提高現金的策略有明確的正向回饋。

`softsign(x) = x / (1 + |x|)` — 有界、無梯度消失問題。

### 3.4 觀測空間（Observation Space）

形狀：`(num_stocks, window_size × num_market_features + 6)`（`enable_sl_features=True` 時再 +3 維 SL 信號）

每支股票的觀測 = 過去 20 天市場特徵（flatten） + 6 個帳戶特徵：

| 帳戶特徵 | 說明 |
|---|---|
| `cash_ratio` | 現金佔比 |
| `total_return` | 本次 episode 累積報酬（clip [-1,1]） |
| `max_drawdown` | 本次 episode 最大回撤 |
| `position[i]` | 此股目前持倉權重 |
| `trade_return[i]` | 此持倉的累積報酬（clip [-1,1]） |
| `holding_period[i] / 100` | 持倉天數（正規化） |

## 4. 交易環境（TaiwanStockEnv）

**檔案**：[`trading_env.py`](trading_env.py)

### 4.1 交易成本模型

```python
COMMISSION_RATE = 0.001425  # 0.1425% 手續費（台灣折扣後約此水準）
TAX_RATE_SELL   = 0.003     # 0.3% 賣出證交稅
SLIPPAGE_RATE   = 0.001     # 0.1% 模擬滑價
```

**每筆交易成本** = `|Δweight| × portfolio_value × (commission + slippage)`，賣出時額外加 `× tax`。

### 4.2 Drawdown 追蹤

```python
if portfolio_value > peak_value:
    peak_value = portfolio_value
max_drawdown = max(max_drawdown, (peak - current) / peak)
```

此 `max_drawdown` 是 episode 內的歷史最高回撤，用於 reward 計算與 Gate 評估。

### 4.3 三種模式

| 模式 | 條件 | 說明 |
|---|---|---|
| **Long-only + Full Stock** | `enable_cash_action=False` | 所有資金分配至股票，無現金空間 |
| **Long-only + Cash** | `enable_cash_action=True` | 最後一個 logit 控制現金比例 |
| **Long/Short + Margin** | `enable_margin_short=True` | Tanh 模式，允許做空，受 `max_leverage` 限制 |

## 7. Walk-Forward 驗證

**檔案**：[`walk_forward.py`](walk_forward.py) + [`research_pipeline.py`](research_pipeline.py)

### 7.1 設計理念

Walk-Forward 是時序驗證的標準方法，避免 look-ahead bias：

```
Period 1 (2024H2):
  Train: 2020-01-01 ~ 2024-06-30  → 訓練模型 A
  Test:  2024-07-01 ~ 2024-12-31  → OOS 評估模型 A

Period 2 (2025H1):
  Train: 2020-01-01 ~ 2024-12-31  → 訓練模型 B（累積更多資料）
  Test:  2025-01-01 ~ 2025-06-30  → OOS 評估模型 B

... (共 4 個 period)
```

每個 period 獨立訓練一個模型，OOS 結果拼接為整段評估曲線。

### 7.2 執行矩陣

```bash
# 單一組合
python walk_forward.py --algo ppo --cash-mode enabled --seeds 42,43,44 --timesteps 300000

# 完整研究矩陣（PPO/SAC × cash on/off）
python walk_forward.py --matrix --seeds 42,43,44 --timesteps 300000
```

### 7.3 結果檔案命名規範

```
results_dir/
  metrics_{algo}_{cash_mode}[_with_features]_wf_seed{N}.json   ← 整體指標
  wf_{algo}_{cash_mode}[_with_features]_model_{period}_seed{N}.zip   ← 每期模型
  walk_forward_{algo}_{cash_mode}[_with_features]_seed{N}.png   ← 績效圖
```

### 7.4 key 指標（metrics JSON）

```json
{
  "algo": "ppo",
```

## 8. Experiment Report 與 Promotion Gate

**檔案**：[`experiment_report.py`](experiment_report.py) + [`promotion_gate.py`](promotion_gate.py)

### 8.1 Promotion Gate 系統

實驗報告後會執行 7 項品質關卡，全通過才能「晉升」到生產環境：

| # | Gate 名稱 | 通過條件 | 說明 |
|---|---|---|---|
| 1 | **Sortino Stability** | seeds ≥ 3，Sortino ≥ 0.8 | 穩定性（跨 seed 不崩） |
| 2 | **Drawdown Gate** | worst-case MDD ≤ 35% | 最大回撤上限（mean + std） |
| 3 | **Cash Behavior** | 非 weak cash | 現金使用行為合理 |
| 4 | **Turnover Gate** | worst-case daily ≤ 10% | 換手率不能太高 |
| 5 | **Period Consistency** | 無極端離群期間 | 各期績效一致（非單期爆發） |
| 6 | **Baseline Comparison** | 打贏 ≥ 50% 基準 | 超越 TWII / 0050 / buy-and-hold |
| 7 | **Ablation (Features)** | Sortino Δ > 0 | overnight feature 確實有貢獻 |
| （可選） | **Stress Testing** | ≥ 70% 情境存活 | 高手續費/滑價下仍 > 0 |

worst-case = mean + std（保守估計最壞情境）。

### 8.2 P5 分析流程

```bash
python p5_analysis.py baseline    # 跑 TWII/0050 buy-and-hold 作為基準線
python p5_analysis.py ablation    # 比較 with/without overnight features
python p5_analysis.py stress      # 4 種成本情境（base/high_fee/high_slippage/worst_case）
python experiment_report.py       # 彙整所有 metrics + 執行 Promotion Gate
```

## 9. 每日交易流程（Daily Trade Runner）

**檔案**：[`daily_trade_runner.py`](daily_trade_runner.py)

### 9.1 完整流程時序

```
06:00  preopen_macro_check.py   → 市場風險評估
         ↓ level = OK / WARN / CRITICAL
06:15  auto_login.py            → 更新 CMoney Cookie
06:30  cmoney_rpa.py --pending-buys  → 執行前一日排程的 BUY 單
         ├─ CRITICAL → 跳過
         └─ WARN → 買入數量減半
09:00  evaluate_portfolio.py    → 載入模型，生成 signal.json
         ↓
       trade_guard.py           → 驗證 signal + 生成 dry-run diff
         ↓ (通過 TTL / AID / 風控)
13:30  cmoney_rpa.py --sell-only → 送出賣單（T+1 模式）
                                   → pending_buys 存至 logs/ 待隔日
         ↓
       Email 通知
```

### 9.2 T+1 模式設計

大富翁為虛擬交易所，結算有延遲。策略上：
- **當日（T）**：送出 SELL 單，等待結算釋放資金
- **隔日（T+1）**：以釋放的資金執行 BUY 單

pending_buys 存檔格式：`logs/pending_buys_{signal_id}_{aid}.json`

### 9.3 Macro Guard 三級制

| Level | 觸發條件 | 動作 |
|---|---|---|
| `OK` | 正常 | 一切照常執行 |
| `WARN` | 市場警訊（VIX 高、外資大賣等） | Pending BUY 數量減半（`--half-buys`） |
| `CRITICAL` | 極端風險（熔斷、崩盤信號） | 跳過所有 BUY 單，保留現金 |

### 10.2 Signal 驗證邏輯

```python
load_signal(signal_path, aid, ttl_seconds=900)
```

1. **AID match**：`signal.aid == CMONEY_AID`（避免送到錯帳號）
2. **TTL check**：`now - created_at ≤ 900 秒`（15分鐘 TTL，避免陳舊信號）
3. **格式驗證**：有 `target_weights` 或 `target_lots`；數值非負
4. **ticker 正規化**：去掉 `.TW` 後綴，只保留代號數字

## 11. Pre-Trade Guard（trade_guard.py）

**檔案**：[`trade_guard.py`](trade_guard.py)

### 11.1 功能

在每次下單前強制執行以下驗證，並生成 dry-run diff：

```
signal.json
    → load_signal()    # TTL / AID / 格式驗證
    → CMoneyRPA.get_account_status()  # 查詢目前庫存
    → build_dry_run_diff()  # 計算需買/需賣的差額
    → evaluate_risk_limits()  # 風控檢查
    → 寫出 trade_guard_diff.json
```

### 11.2 風控限制（可配置）

```python
RiskLimits:
  max_single_weight = 0.35    # 單一股票最大倉位 35%
  max_total_exposure = 1.0    # 總曝險不超過 100%（不允許借貸）
  max_leverage = 2.0          # 最大槓桿（margin 模式）
  max_turnover = 0.30         # 單次換手率上限 30%
```

### 11.3 dry-run diff 格式

```json
{
  "signal_id": "...",
  "current_lots": {"2330": 5, "2454": 3},
  "target_lots":  {"2330": 8, "2317": 2},
  "plan": {
    "buys":  {"2330": 3, "2317": 2},
    "sells": {"2454": 3}
  },
  "net_buy_lots": 5,
  "net_sell_lots": 3,
  "risk_checks": { "passed": true, "reasons": [] }
}
```

## 16. 常見問題與除錯

### Q1：Promotion Gate 的 Drawdown Gate 失敗

**可能原因**：
1. 訓練 timesteps 不足（10K 快速訓練 MDD 偏高）
2. OOS 期間包含大幅市場下跌（2024年4月/2025年4月等）

**解法**：
- 增加 timesteps 到 300K+（`--timesteps 300000`）
- R4 已調整 reward（`trading_env.py`）：`LAMBDA_DRAWDOWN` 0.3→0.8，緩衝 5%→3%，regime 門檻 10%→8%（係數 0.5→1.0），新增防禦現金獎勵（`LAMBDA_CASH_DEFENSIVE=0.2`）
- 重新 walk-forward 訓練後方可驗證效果（300K × 3 seeds）

---

### Q2：Cookie 失效

**症狀**：`PermissionError: Cookie 已經失效`

**解法**：
1. 用瀏覽器登入 cmoney.tw
2. F12 → Network → 任意請求 → Request Headers → Cookie
3. 複製完整 Cookie 字串更新 `.env` 的 `CMONEY_COOKIE`

---

### Q3：signal.json TTL 失效

**症狀**：`SignalError: Signal expired`

**原因**：`evaluate_portfolio.py` 產生 signal 後超過 900 秒才執行下單

**解法**：確保 Step 1（evaluate）和 Step 2（RPA）在同一個排程內連續執行，或增加 `.env` 的 `SIGNAL_TTL_SECONDS`。

---

### Q4：Walk-Forward 某期被跳過

**症狀**：`Skip 2026H1: test_start 2026-01-01 is after today`

**原因**：正常行為，`clamp_periods()` 會略過未來期間

**說明**：2026H1 的 test 期間（2026上半年）目前仍在進行中，會被 clamp 到今天，因此有效測試資料會隨時間增加。

---

### Q5：ablation n=1 樣本不足

**症狀**：`[WARN] No matching seed pairs found for ablation`

**原因**：`_with_features` 系列只有 seed42，比對時找不到 seed43/44 的 with-features 結果

**解法（N5）**：
```bash
python walk_forward.py --algo ppo --cash-mode enabled --seeds 43,44 --timesteps 300000 ^
  --overnight-feature-path capital_flow_analysis/data/overnight_gap_features_1d.csv --overwrite
```
